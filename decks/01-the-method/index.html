Why this exists

Because we have a lot of stuff to build this summer.

This is the first of three slide decks that will guide you through a structured methodology for productive building with an LLM assistant.

We're building HTML-based presentations that teachers can launch in a browser window with zero dependencies. LLMs are great at this, but you have to guide them well. This presentation was built with the same guidance methodology — llm-dev:cycle — that you'll be learning to use here.

From topic to deck

Set the bar plainly: solo authorship. The audience is mixed — some code, some don't. The honest message is that the skill being taught is directing the model, not memorizing CSS. The cycle is the structure that makes that reliable instead of a chaotic chat.

The offline constraint is the reason we use plain HTML rather than a hosted tool — say so, because it justifies everything that follows.

03
The shape of the work

The llm-dev:cycle is an iterative loop with 6 phases

The cycle: Review → Brainstorm → Research → Plan → Execute → Verify

Each of the six phases is defined by the llm-dev:cycle/SKILL.md file, with further details in cycle/references/phase-details.md. Loading these two short files into the context of an LLM conversation gives your LLM everything it needs to walk through each phase of the cycle automatically.

  1. Review / Reflect — load context
  2. Brainstorm — decide what to build
  3. Research — pick the tools, verify them
  4. Plan — an ordered build sequence
  5. Execute — build it, test as you go
  6. Verify — prove it actually works

Then you iterate: each pass is smaller and sharper than the last.

Six phases, one loop

The key idea: the value is in making decisions before committing to code. Each phase has a gate — you approve before moving on. Nothing here is rigid; you can revise a phase or jump back. Stress that the markdown artifacts are the point: they make the reasoning reviewable and reusable.

04
Phase 1

Review / Reflect

On the first llm-dev:cycle iteration, review and load context to prime the LLM. On subsequent cycle iterations, reflect on what's been built and guide the next cycle.

This is where you give the LLM background about what you want to build — a presentation outline, reference materials, an overarching goal to achieve, design templates. Higher input context quality in this phase will translate to higher output quality of the final result.

Treat the LLM as a highly competent design partner, and bring it up to speed on everything you know about what you want to build.

Review / Reflect

The discipline here is narrowing, not expanding — eliminate options, surface gaps. For a deck, the audience + delivery + runtime constraints decide almost everything downstream (e.g. "offline classroom" forced vanilla HTML).

05
Phase 2

Brainstorm

Decide what to build — through conversation, not a menu.

Explore the idea with the model: what's the simplest version, what's the narrative arc, what's in and out of scope. Converge on one approach and write down the alternatives you rejected and why.

For a presentation: the story comes first. Decide the arc before any slide exists.

Brainstorm

It's a dialogue. The model should ask questions and push back, not just produce options. The output is a scoped concept with explicit boundaries — what you will and won't build. Rejecting things on purpose is half the value.

06
Phase 3

Research

Pick the tools — and verify they're real.

Choose the simplest thing that meets the requirements, and check it actually works before committing. Confirm what you can lift from an existing example rather than building from scratch.

For a presentation: usually "vanilla HTML/CSS/JS, reuse a known-good template." Verify it runs offline.

Research

The honest note: models sometimes recommend packages that don't exist or are deprecated. Verify. For our decks this phase mostly confirms the template lifts cleanly and that nothing reaches for the network.

07
Phase 4

Plan

Turn research into an ordered build sequence.

What gets built first, second, third — with acceptance criteria: specific, testable conditions that define "done." A good plan is self-contained; you could hand it to someone else and they'd know what to do.

For a presentation: build the template first, then slides; "done" = plays start to finish, offline, no console errors.

Plan

If a step would take more than an hour, break it down. The acceptance criteria are what Verify checks against later — write them as pass/fail, not vibes.

08
Phase 5

Execute

Build it — and test each piece as you go.

Work through the plan step by step. Don't wait until the end to check things; smoke-test as each slide or component lands. When something breaks, fix it or adjust the plan.

For a presentation: open it in the browser after each slide. Catch the broken image path now, not at the podium.

Execute

"Test early and often" is the whole lesson. Describe errors precisely to the model — "this returns X, I expected Y." Read the code it writes; understand it enough to explain it.

09
Phase 6

Verify

Prove it works — assume it doesn't until you've checked.

Execute reports on its own work, so Verify is the adversarial counterpart. Walk every acceptance criterion and confirm pass or fail with evidence, not assertions.

For a presentation: open it offline, click through every slide, watch the console, confirm no network requests.

Verify

Evidence before assertions. "It works" is not verification; "I opened it via file://, clicked all 11 slides, console was clean, Network tab showed zero requests" is. If it fails, you don't mark it done — you fix or carry it forward.

10
Driving the cycle

Gates, iteration, and parallel work

Gates
Approve between phases
Iterate
Each pass smaller
Parallel
Fan out independent work
  • Gates are guided, not rigid. At each phase you say "next," "revise," "skip," or "back" — you stay in control.
  • Iteration is the point. Ship a small slice, learn, then the next pass reflects on what happened. This deck was iteration 1 of its project.
  • Parallel dispatch when work is independent — e.g. summarizing several reference docs at once. The skill suggests it; you decide.

Driving the cycle

This is the "how it feels to use it" slide. The model proposes; you approve. Emphasize that skipping and revising are normal — not every phase needs the same weight. Iteration keeps scope honest.
