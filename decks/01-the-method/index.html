02
Why this exists

LLMs can help us build amazing things this summer.

To consistently produce amazing results, though, LLMs need consistent guidance.

llm-dev:cycle is a methodology for providing consistent guidance to LLM assistants. In fact, the HTML slide deck you are reading now was built following this methodology.

In the following slides, you will learn how to use llm-dev:cycle to build your own HTML slide decks (and other stuff) that teachers can easily launch in a browser window.

If you'd like, you can install the entire llm-dev plugin for Claude Code.
All you really need to get started are the skill files from the ASWP repo, though.

From topic to deck

Set the bar plainly: solo authorship. The audience is mixed — some code, some don't. The honest message is that the skill being taught is directing the model, not memorizing CSS. The cycle is the structure that makes that reliable instead of a chaotic chat.

The offline constraint is the reason we use plain HTML rather than a hosted tool — say so, because it justifies everything that follows.

03
The shape of the work

The llm-dev:cycle is an iterative loop with 6 phases

The loop: Review → Brainstorm → Research → Plan → Execute → Verify → (repeat as necessary)

All six phases are defined by SKILL.md and references/phase-details.md in the ASWP repo.


Loading these two short files into the context of an LLM session provides everything needed to walk through and / or iterate the cycle automatically. Here's a brief description of each phase:


  1. Review / Reflect: Load context for a new loop by reviewing requirements or reflecting on past progress.
  2. Brainstorm: Ideate, discuss, and decide conceptually before actually building the thing.
  3. Research: Select and verify the languages, tools, tech stack, etc. that your thing will be built from.
  4. Plan: Create an ordered, step by step sequence to guide building and testing of your thing.
  5. Execute: Follow your plan to actually build and test the thing.
  6. Verify: Spend more time with the thing you've built. Decide whether you're happy with it, and why.

If your thing needs more work: iterate. Ideally, each loop will be smaller and sharper than the last.

Six phases, one loop

The key idea: the value is in making decisions before committing to code. Each phase has a gate — you approve before moving on. Nothing here is rigid; you can revise a phase or jump back. Stress that the markdown artifacts are the point: they make the reasoning reviewable and reusable.

The shape of the work

The llm-dev:cycle creates an artifact for each phase

The loop: Review → Brainstorm → Research → Plan → Execute → Verify → (repeat as necessary)

This helps track progress and creates an easy entry point for future sessions.

Each iteration is its own folder; each phase leaves a short markdown artifact behind. The thinking is written down before — and as — the thing gets built.

An artifact for each phase

These markdown files are the durable record of the work: what was decided, what was tried, what was verified. They make the reasoning reviewable and reusable, and they're what a later iteration reflects on. This very deck was built this way — its own workspace/cycle/ folder looks just like this.

05
Phase 1

Review / Reflect

Bring the LLM up to speed.

  • On first llm-dev:cycle iteration: Review and load context to prime the LLM.
  • On subsequent llm-dev:cycle iterations: Reflect on what's been built and guide the next cycle.

Treat the LLM as a highly competent design partner. Bring it up to speed by providing necessary background about what you want to build. You could include:

  • markdown files
  • rough draft .pptx presentation files
  • other reference materials (URLs, research papers, etc.)
  • a list of goals to achieve
  • a design template or description, etc.

This phase is about building a foundation for your project: review.md. In the next phase, you'll ideate and elaborate.

Review / Reflect

The discipline here is narrowing, not expanding — eliminate options, surface gaps. For a deck, the audience + delivery + runtime constraints decide almost everything downstream (e.g. "offline classroom" forced vanilla HTML).

06
Phase 2

Brainstorm

Explore ideas with the LLM through conversation.

Get creative. Ask questions, and ask the LLM to ask you questions. You might think about the following: what's the simplest version of what you want to create? What's the narrative arc or story you want to tell? What's in and out of scope? What might be reserved for future iterations? The LLM will help you converge on one approach, and then write down the alternatives you considered / rejected, and why.

This phase is about choosing a conceptual path forward, which the LLM records in brainstorm.md. In the next phase, you'll consider and research technical implementation options.

Brainstorm

It's a dialogue. The model should ask questions and push back, not just produce options. The output is a scoped concept with explicit boundaries — what you will and won't build. Rejecting things on purpose is half the value.

07
Phase 3

Research

Work with the LLM to identify and verify your tech stack.

For our purposes, vanilla HTML and JS will be sufficient and should accommodate our users' technical limitations (Chromebook-only, limited network access, etc.). Depending on the outputs of the Brainstorming phase, the LLM may research several implementation options and present them to you. Trust your LLM to give good advice, but be careful not to lazily outsource your own judgement too much : )

This phase is about settling on a technical implementation approach, which the LLM will record in research.md. In the next phase, your LLM will help you create a detailed step-by-step implementation plan.

Research

The honest note: models sometimes recommend packages that don't exist or are deprecated. Verify. For our decks this phase mostly confirms the template lifts cleanly and that nothing reaches for the network.

08
Phase 4

Plan

Turn research into an ordered build sequence.

What gets built first, second, third — with acceptance criteria: specific, testable conditions that define "done." A good plan is self-contained; you could hand it to someone else and they'd know what to do. The LLM will record this as plan.md; your job is to review and verify / modify if needed before moving on. Make sure that your plan includes sufficient testing along the way to verify build quality — when in doubt, ask your LLM whether the plan includes sufficient tests for the build.

Once your Plan is solid, you'll move on to the Execution phase.

Plan

If a step would take more than an hour, break it down. The acceptance criteria are what Verify checks against later — write them as pass/fail, not vibes.

09
Phase 5

Execute

Build according to plan.md, and test each piece as you go.

Your LLM will work through plan.md step by step, testing as it builds. It may ask whether to execute the plan in-line or via subagent-driven development; either of these options is probably fine, but there are trade-offs: subagent-driven dev will better preserve the main session context window, but in-line dev may better preserve the original intent of your build.

Your LLM will record the results of the Execution phase in execute.md. Once complete, you'll move to the Verify phase for hands-on build validation.

Execute

"Test early and often" is the whole lesson. Describe errors precisely to the model — "this returns X, I expected Y." Read the code it writes; understand it enough to explain it.

10
Phase 6

Verify

Prove it works — assume it doesn't until you've checked.

Execute reports on its own work, so Verify is the adversarial counterpart. Walk every acceptance criterion and confirm pass or fail with evidence, not assertions.

The LLM records the results in verify.md. For a presentation: open it offline, click through every slide, watch the console, confirm no network requests.

Verify

Evidence before assertions. "It works" is not verification; "I opened it via file://, clicked all 11 slides, console was clean, Network tab showed zero requests" is. If it fails, you don't mark it done — you fix or carry it forward.

11
Driving the cycle

Gates, iteration, and parallel work

Gates
Approve between phases
Iterate
Each pass smaller
Parallel
Fan out independent work
  • Gates are guided, not rigid. At each phase you say "next," "revise," "skip," or "back" — you stay in control.
  • Iteration is the point. Ship a small slice, learn, then the next pass reflects on what happened. This deck was iteration 1 of its project.
  • Parallel dispatch when work is independent — e.g. summarizing several reference docs at once. The skill suggests it; you decide.

Driving the cycle

This is the "how it feels to use it" slide. The model proposes; you approve. Emphasize that skipping and revising are normal — not every phase needs the same weight. Iteration keeps scope honest.
