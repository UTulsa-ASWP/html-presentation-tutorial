The story

What you're about to see

Over two work sessions, an instructor and an LLM turned a topic — "explain differential privacy" — into a polished, interactive, offline web deck.

The build followed the same six phases from Deck 1. Along the way: a sharp design pivot, a "no dependencies" decision that paid off, a mid-build refactor, a bug caught by testing, and a same-day deploy. We'll walk each phase, then highlight the moments that mattered most.

Phase 1 · Review

The constraint that shaped everything

The very first phase pinned down the hard limit the whole build had to respect:

"Must be ready by tomorrow… Single-session build."

A same-day deadline with a live, in-class demo and no server to lean on. That one constraint quietly decided the tool, the scope, and the format before a line of code existed.

04
Phase 2 · Brainstorm

The pivot: explain, don't crowdsource

The tempting idea was a participatory, multi-device demo. Brainstorm rejected it in favor of an instructor-driven explainer:

"A well-designed interactive explainer can drive discussion as effectively as participation."

This was the foundational key moment: the narrative and pedagogy were designed first, then the interactive demos were mapped onto that arc — not the other way around.

Read the full Brainstorm doc →

Brainstorm

Story-first is the lesson. The 8-act arc (Sweeney's $20 re-identification, the Spartacus k-anonymity analogy, Dwork's 2006 "revolution," real deployments, minority erasure as the dark side) came before any slide. Rejecting the participatory idea was a deliberate scope-narrowing for a same-day build.

05
Phase 3 · Research

Zero dependencies, on purpose

"Vanilla HTML/CSS/JavaScript in a single self-contained file. No framework, no build step, no external JS libraries." — Zero dependency risk.

React, D3, Chart.js — all considered and rejected. The payoff: sophisticated interactivity (Laplace-noise sampling, live SVG histograms, sliders) built in pure client-side JavaScript, with nothing to break between now and class tomorrow.

Read the full Research doc →

Research

This is the decision that makes the artifact offline — the same reason our tutorial decks are vanilla too. Note that "simplest tool that meets the requirements" beat the reflex to reach for a framework.

06
Phase 4 · Plan

An ordered build, with room to split

Plan laid out the file structure and an ordered sequence: scaffold first, then build the narrative acts in order, then the interactive exhibits, then deploy.

It started as a single self-contained file — but planned for a clean structure that could split as complexity grew. That foresight set up the refactor we'll see in Execute.

Acceptance criteria were written as pass/fail checks — the same ones Verify would later confirm.

Read the full Plan doc →

Plan

The plan is self-contained: someone else could execute it. The split-file foresight is the link to the next slide's refactor — plans anticipate where the pressure will be.

07
Phase 5 · Execute

Build, test, fix — in the open

Roughly 1,766 lines came together, with every interactive exhibit tested in the browser as it landed. Testing immediately paid off:

The epsilon slider's range was too wide. Fix: "Narrowed to min=-10 max=13, giving a meaningful range of ε ≈ 0.1 to ε ≈ 20."

The demo was complete but pedagogically broken until that fix made the privacy-vs-utility tradeoff actually teachable.

See it live: open the explainer and try the epsilon slider and the linkage-attack demo.

Read the full Execute doc →

Execute

"Test early and often" is the whole lesson — the bug was caught because each interactive was exercised as it was built, not at the end. Have them open the slider and re-roll the noise a few times to feel the randomness.

08
Phase 6 · Verify

Proven by hand, with evidence

"Status: COMPLETE — manually verified by user… hands-on in Chrome on macOS."

Every slide and all four interactive exhibits were walked through by a human in the browser. Two items — Safari and public deploy — were consciously deferred as low-risk, not silently skipped.

Read the full Verify doc →

Verify

Evidence over assertion: "verified in Chrome on macOS, all four interactives exercised" beats "looks done." Deferring Safari/deploy explicitly is honest scope management, not cutting corners.

09
Key moment

One file became three — mid-build

As slides multiplied and interactivity grew, the monolithic HTML file was refactored on the fly into three:

.html
content
.css
styling
.js
interactivity

It was safe because the IDs weren't referenced in JS and the slide counter auto-computes from slideElements.length — so there were no hardcoded numbers to break. Structure emerged exactly when complexity demanded it.

The refactor

This is the "refactoring emerges from the cycle" lesson made concrete — it wasn't scheduled, it was triggered by the code signaling it was time. The same three-file split is what our starter template uses.

10
Key moment

How it really went

The transcripts show the human side of the build — direction given in real time:

"…we will just start editing the slide deck because I don't really have time to go back through another iteration."

And decisiveness — a one-word "B" chose the stronger slide order (insert the k-anonymity demo, push the homogeneity reveal down as the "even this still fails" twist). A real gotcha, too: a deploy-then-sanitize git reset --hard wiped uncommitted edits, recovered from the already deployed copy.

How it really went

Humanize it: the process bends to real constraints (time pressure → skip a formal iteration), decisions can be one word, and tools have sharp edges (the reset mishap). None of this is failure — it's what real builds look like.
