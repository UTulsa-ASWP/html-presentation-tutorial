Phase 1 · Review

The constraint that shaped everything

The very first phase pinned down the hard limit the whole build had to respect:

"Must be ready by tomorrow… Single-session build."

A same-day deadline with a live, in-class demo and no server to lean on. That one constraint quietly decided the tool, the scope, and the format before a line of code existed.

Phase 1: Review

The teaching point: naming the hard constraint early is what makes every later decision obvious. Here "same-day + offline + live demo" does all the work — it quietly settles the tool, the scope, and the format before any code exists.

When you run your own cycle, this is the phase to pin down who the audience is, how the result is delivered, and what it has to run on. Those constraints cascade through everything downstream.

04
Phase 2 · Brainstorm

The pivot: explain, don't crowdsource

The tempting idea was a participatory, multi-device demo. Brainstorm rejected it in favor of an instructor-driven explainer:

"A well-designed interactive explainer can drive discussion as effectively as participation."

This was the foundational key moment: the narrative and pedagogy were designed first, then the interactive demos were mapped onto that arc — not the other way around.

Read the full Brainstorm doc →

Phase 2: Brainstorm

Story-first is the lesson. The eight-act arc — Sweeney's $20 re-identification, the Spartacus k-anonymity analogy, Dwork's 2006 "revolution," real deployments, and minority erasure as the dark side — was designed here, in Brainstorm, before a line of HTML existed.

Just as important was what got rejected: the tempting participatory, multi-device demo was deliberately cut to keep a same-day build achievable. Deciding what you won't build is half the value of this phase.

05
Phase 3 · Research

Zero dependencies, on purpose

"Vanilla HTML/CSS/JavaScript in a single self-contained file. No framework, no build step, no external JS libraries." — Zero dependency risk.

React, D3, Chart.js — all considered and rejected. The payoff: sophisticated interactivity (Laplace-noise sampling, live SVG histograms, sliders) built in pure client-side JavaScript, with nothing to break between now and class tomorrow.

Read the full Research doc →

Phase 3: Research

This is the decision that makes the artifact offline — the same reason our tutorial decks are vanilla too. The reflex is to reach for a framework (React, D3, Chart.js); Research rejected all of them in favor of the simplest tool that meets the requirements.

The payoff is sophisticated interactivity — Laplace-noise sampling, live SVG histograms, sliders — built in pure client-side JavaScript, with nothing that can break between now and class tomorrow.

Phase 4 · Plan

An ordered build, with room to split

Plan laid out the file structure and an ordered sequence: scaffold first, then build the narrative acts in order, then the interactive exhibits, then deploy.

It started as a single self-contained file — but planned for a clean structure that could split as complexity grew. That foresight set up the refactor we'll see in Execute.

Acceptance criteria were written as pass/fail checks — the same ones Verify would later confirm.

Phase 4: Plan

A good plan is self-contained — someone else could execute it without being in the room. Two things this one got right: an ordered build sequence (scaffold → narrative acts → interactive exhibits → deploy), and acceptance criteria written as pass/fail checks, which are exactly what Verify confirms later.

The "plan for a structure that can split as complexity grows" foresight is the direct setup for the refactor on slide 9 — good plans anticipate where the pressure will land.

07
Phase 5 · Execute

Build, test, fix — in the open

Roughly 1,766 lines came together, with every interactive exhibit tested in the browser as it landed. Testing immediately paid off:

The epsilon slider's range was too wide. Fix: "Narrowed to min=-10 max=13, giving a meaningful range of ε ≈ 0.1 to ε ≈ 20."

The demo was complete but pedagogically broken until that fix made the privacy-vs-utility tradeoff actually teachable.

See it live: jump straight to the epsilon slider in the explainer and re-roll the noise to feel the fix.

Read the full Execute doc →

Phase 5: Execute

"Test early and often" is the whole lesson. The epsilon-slider bug was caught only because each interactive was exercised in the browser as it landed, not at the end — and until that fix the demo was complete but pedagogically broken, since the privacy-vs-utility tradeoff wasn't legible.

The slide links straight to the fixed slider (#slide-13); re-roll the noise a few times to feel both the randomness and why the narrowed range matters.

Phase 6 · Verify

Proven by hand, with evidence

"Status: COMPLETE — manually verified by user… hands-on in Chrome on macOS."

Phase 6: Verify

Evidence over assertion: "verified hands-on in Chrome on macOS, all four interactives exercised" beats "looks done." Every slide and exhibit was walked through by a human in the browser.

The honest part worth highlighting: Safari and a public deploy were consciously deferred as low-risk and recorded as known gaps, rather than silently skipped. That explicit scope management is what separates verification from wishful thinking.

09
Key moment

One file became three — mid-build

As slides multiplied and interactivity grew, the monolithic HTML file was refactored on the fly into three:

.html
content
.css
styling
.js
interactivity

It was safe because the IDs weren't referenced in JS and the slide counter auto-computes from slideElements.length — so there were no hardcoded numbers to break. Structure emerged exactly when complexity demanded it.

The refactor

This is "refactoring emerges from the cycle" made concrete. The one-file-to-three split wasn't scheduled — it was triggered when the growing code signaled it was time.

It was safe specifically because the slide IDs weren't referenced in JS and the counter auto-computes from slideElements.length, so there were no hardcoded numbers to break. That same three-file split (.html / .css / .js) is what our starter template ships with.

10
Key moment

How it really went

The transcripts show the human side of the build — direction given in real time:

"…we will just start editing the slide deck because I don't really have time to go back through another iteration."

And decisiveness — a one-word "B" chose the stronger slide order (insert the k-anonymity demo, push the homogeneity reveal down as the "even this still fails" twist). A real gotcha, too: a deploy-then-sanitize git reset --hard wiped uncommitted edits, recovered from the already deployed copy.

How it really went

Humanize the process: it bends to real constraints. Time pressure led to skipping a formal iteration and editing the deck directly; a decision can be a single keystroke ("B" chose the stronger slide order); and tools have sharp edges — a deploy-then-sanitize git reset --hard wiped uncommitted edits, recovered only from the already-deployed copy.

None of this is failure. It's what real builds look like, and the transcripts preserve it honestly.
